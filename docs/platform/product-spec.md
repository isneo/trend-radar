## §1 问题陈述

AI 从业者（研发、产品经理、独立研究者）每天消耗 30-90 分钟在信息源轮询：
Twitter / X、HackerNews、Reddit r/LocalLLaMA、Arxiv、各家 AI 公司博客、GitHub Trending、
国内公众号、即刻、V2EX。这种消费模式有三个典型问题场景：

**场景 A：信号淹没在噪声里**
研发 Alice 想追踪"有没有新开源的 agent 框架"。她刷 Twitter 40 分钟，只刷到 2 条相关，
其他全是段子、招聘、自我营销。无法判断"今天没看到"是"真没有"还是"被算法埋了"。

**场景 B：语言壁垒**
PM Bob 英文阅读能力一般，Arxiv 每天 100+ 篇 AI 论文，摘要全英文。看不动 → 放弃 →
信息落后团队里会读英文的同事 1-2 天。

**场景 C：分散与去重**
同一条新闻（例如 Anthropic 发布 Claude 某版本）会在 10 个渠道以不同标题出现。
Carol 为了全面没漏掉，订了 5 个 RSS + 2 个群，结果每天被同一件事推送 7-8 次，
反而更焦虑。

**当前替代方案的缺陷**：

| 方案 | 缺陷 |
|---|---|
| 手动刷 Twitter / HN | 噪声大、有算法偏差、读的都是别人让你读的 |
| RSS 阅读器（Feedly / Reeder） | 重复、没摘要、要自己挑源、外语内容读不动 |
| 即刻 / 小报童 | 中文为主，AI 深度内容少，不能按关键词精准订阅 |
| NotebookLM / Perplexity Daily | 要主动问，不推送；不按兴趣 |
| 公众号订阅 | 有筛选偏差、滞后 1-3 天、不抓一手 |

## §2 目标用户

### Primary：AI 研发工程师

| 维度 | 画像 |
|---|---|
| 角色 | 大厂 AI infra / 模型工程师，或独立 AI 开发者 |
| 核心痛点 | 追踪论文 / 开源 / 模型发布，FOMO 新技术 |
| 当前替代方案 | Twitter + HN + Arxiv-sanity + GitHub star 通知 |
| 为什么不满意 | 分散、英文多、无法按细分方向过滤（如"只要 agent 框架"） |
| 典型上下文 | 每天通勤 / 午休 / 睡前各 10 分钟，用手机 |
| 为此愿意付费 | 中等。可接受 10-30 元/月，前提是帮他每天省 30+ 分钟 |

### Secondary：AI PM

| 维度 | 画像 |
|---|---|
| 角色 | 互联网公司 AI 产品经理 |
| 核心痛点 | 追踪竞品动态、赛道新玩家、行业大新闻 |
| 当前替代方案 | 同行群、即刻、竞品官网定期查 |
| 为什么不满意 | 群里混 + 滞后，即刻 AI 深度内容不够 |
| 典型上下文 | 工作时段碎片时间，用电脑 + 手机 |
| 为此愿意付费 | 中等偏低。20 元/月可接受。 |

### 明确排除的人群

- **通用新闻消费者**：要看社会/政治/娱乐新闻的 → 即刻更合适
- **传统 RSS 重度用户**：自己精心维护 200 个源的 → Feedly 已经够用
- **既不用 TG 又不用飞书的用户**：V1 只有这两个渠道，其他（微信 / 邮件 / Web）看 V2/V3
- **要 Web UI 强交互的**：V1 是推送为主，仅提供极简登记表单，不提供完整管理后台

### §2.3 用户的渠道偏好

| 用户类型 | 首选**接收**渠道 | 原因 | V1 支持 |
|---|---|---|---|
| 境外 AI 从业者 / 国内偏技术圈独立开发者 | Telegram（个人） | 已常用，支持 rich bot 交互 | ✅ |
| 国内企业 / 团队 / 公司内部分享 | 飞书群 webhook | 办公环境已在飞书，群就是现成的分发点 | ✅ |
| 国内只用微信的个人 | 微信 | — | ❌ V3 |
| 喜欢邮件 digest | Email | — | ❌ V2 末期 |

**V1 的控制面统一放在 TG**（重要设计决策，见架构文档 ADR-004）：
- 即便只想让消息落到飞书群，**订阅管理本身（添加关键词 / 删除 / 暂停 / 查看）必须通过 TG 完成**
- TG 作为"遥控器"，Feishu 作为"屏幕"
- V1 内测对象都是技术圈朋友（5-10 人），TG 门槛不是问题
- V2 若接入非技术 Feishu 管理员，再补 Web form（1.5 天工作量）

两种接收路径的数据模型统一：`Subscription(user_id, keywords[], delivery_channels[])`，一个 TG 用户可以同时绑定多个 Feishu 群作为二级投递目标。

## §3 价值主张与差异化

### 核心价值主张

**把每天 30-60 分钟刷 AI 信息的时间，压缩到 3-5 分钟读一次推送。**

达成方式：
1. **策展信源**：我们精选源，用户不用自己挑
2. **关键词过滤**：只推你订阅的方向
3. **智能摘要**：外语 → 中文、长文 → 3 句话、论文 → 一句话要点

### 差异化（3 条，每条附竞品对照）

#### 差异化 A：策展高质量信源（vs 让你自己加 RSS）

**做什么**：编辑团队（V1 由创始人亲自维护）精选 30-50 个信源，分类为：
`论文`、`开源`、`产品发布`、`融资`、`深度文章`、`国内动态`。新源由用户提名 + 编辑筛选加入。

**为什么 Feedly / Reeder 做不到**：
这些是通用 RSS 客户端，让用户自己挑。问题是：
- 普通用户不知道哪些源值得订（需要信息素养）
- 好源混杂噪声（如官博里有一半是招聘）
- 没有"分类组"概念（全靠用户自己分文件夹）

我们的定位不是 RSS 客户端，是**"AI 方向的编辑精选 + 推送引擎"**。

#### 差异化 B：外语内容自动中文化（vs 让你自己读英文）

**做什么**：所有英文源（Arxiv 摘要、官博、Reddit）自动 LLM 翻译 + 摘要为中文，
原文链接保留。用户在 TG 里看到的是：

```
[Arxiv] 2026-04-23
Agent Planning via RL with Human Feedback
🇨🇳 摘要：这篇论文在 ReAct 基础上引入 RLHF 来训练 planner……
🔗 https://arxiv.org/abs/xxxx.xxxxx
```

**为什么即刻 / 公众号做不到**：
- 即刻是 UGC，翻译要靠博主主动搬运，覆盖度不稳定且有滞后
- 公众号对 Arxiv / 英文博客的覆盖偏点状
- 没有 LLM 翻译 + 摘要的系统化产能

#### 差异化 C：论文一句话摘要（vs 读长 abstract）

**做什么**：Arxiv 每天新增 AI 论文的 abstract 再次被 LLM 浓缩成一句话"这篇在解决什么、
用什么方法、结论是什么"。用户看到一句话判断要不要点链接读 abstract、再决定要不要读全文。
三级漏斗：**一句话 → abstract → 全文**。

**为什么 Arxiv-sanity / Daily Arxiv Newsletter 做不到**：
- Arxiv-sanity 基于点击率推，不做摘要
- Newsletter 是 abstract 原文，中文用户体验差
- 我们加了"二次浓缩 + 中文 + 按订阅关键词过滤"

## §4 V1 功能范围（目标 2-4 周本地跑通）

### 功能清单

| # | 功能 | 验收信号 | 工作量 |
|---|---|---|---|
| F1 | 用户通过 TG Bot `/start` 注册 | 手机发 /start，bot 回欢迎，DB 多一条 user | 0.5 天 |
| F2 | `/subscribe <关键词>` 添加订阅 | 发 `/subscribe agent 框架`，/list 能看到 | 0.5 天 |
| F3 | `/list` 查看订阅 | 返回当前所有关键词 | 0.2 天 |
| F4 | `/unsubscribe <id>` 删订阅 | 删除后 /list 消失 | 0.3 天 |
| F5 | Celery beat 每小时定时爬 | 日志有 `[CRAWL] 00:00 started` | 0.5 天 |
| F6 | 爬取内容按用户订阅 contains 过滤 | 日志显示 `user#1 matched 3 items` | 1 天 |
| F7 | 匹配项通过 TG 推送到用户 | 手机收到消息 | 0.5 天 |
| F8 | delivery_log 去重（同 user+item+channel 不重推） | 手动连跑 2 次，第 2 次不重复推 | 0.5 天 |
| F9 | `/now` 手动触发一次爬取（用户侧） | 发 /now，15 秒内收到推送（若有匹配） | 0.5 天 |
| F10 | 策展信源池：复用现有 trendradar 30+ 源 | config/config.yaml 保留 | 0 天（已有） |
| F11 | 飞书自定义 webhook 推送（富文本卡片） | 群里收到匹配消息 | 0.5 天（复用 trendradar 现有 Feishu 代码） |
| F12 | TG 命令 `/add_feishu_group <webhook_url> <关键词>` | 发命令后 DB 有记录；该群收到欢迎消息确认 | 0.5 天 |
| F13 | TG 命令 `/my_feishu_groups` / `/remove_feishu_group <id>` | 列出 / 删除当前 TG 用户名下的 Feishu 群 | 0.3 天 |
| F14 | 数据模型：Subscription 支持多 delivery target（TG + N 个 Feishu 群） | 一条订阅可同时推 TG 和多个群 | 0.3 天 |

**合计工作量**：5-6 天纯编码 + 2-3 天联调测试 = **本地 V1 在 8-10 天内完成**。

**v0.3 注意**：F12/F13 用 TG 命令实现，不做 Web form。这样 V1 无 Web 前端代码、无 CORS、无 Turnstile、无 magic link。

V1 **不包含**：
- LLM 翻译 / 摘要（V2 再加）
- 共享信源策展流程（V1 直接用现有 trendradar 的信源表）
- Arxiv 专项爬取（V2）
- 付费 / 限额
- **任何 Web UI**（v0.3 决策：控制面全在 TG，V2 若需要再补 Web form）
- 其他渠道（邮件 / 微信 / 企业微信）
- 飞书 Bot 应用（需要企业认证，走 webhook 就够）

### V1 最小跑通的 End-to-End 场景

**场景 A：TG 个人用户**
```
neo 在 TG 上：/start
  → Bot 回复欢迎语
neo：/subscribe 大模型
  → "已订阅：大模型"
[Celery beat 触发下一次爬取]
  → 爬取返回 50 条新闻，其中 3 条标题 contains "大模型"
  → Bot 推送给 neo 这 3 条
neo 在 TG 上看到推送
neo：/unsubscribe 1 → 删除
```

**场景 B：飞书群通过 TG 管理员绑定（v0.3）**
```
Carol（技术圈朋友）在公司飞书里建群「AI-infra 技术前沿」
  → 群设置 → 机器人 → 添加自定义机器人 → 复制 webhook URL
切到 TG，给 TrendRadar Bot 发：
  /add_feishu_group https://open.feishu.cn/open-apis/bot/v2/hook/xxx 大模型, agent, 推理加速
TG Bot 回：
  "已绑定，并已发测试消息到群「AI-infra 技术前沿」"
  飞书群里同时收到欢迎卡片，确认订阅生效
[Celery beat 触发下一次爬取]
  → 匹配到 3 条新闻
  → 飞书群收到 3 条富文本卡片（标题 + 摘要 + 原文链接）
Carol 想改订阅 → 回 TG 发 /my_feishu_groups → 看到群列表 → /edit_feishu_group 1 新关键词
```

**"成功"标准**：上述两个场景在本地端到端跑通，`delivery_log` 按 `(subscription_id, item_fingerprint, delivery_target)` 去重，两次运行不重推。

## §5 V2 / V3 路线

### V2（上线后 1-3 个月）

**主题：差异化能力落地**

| 功能 | 为什么放 V2 |
|---|---|
| 英文内容 LLM 翻译 + 中文摘要 | 差异化 B 的核心，需要稳定的 AI 调用，V1 先验证推送链路 |
| Arxiv 每日 AI 论文一句话摘要 | 差异化 C 的核心 |
| 跨源去重（同一新闻不同标题判重） | 有一定数据后才看得出重复模式 |
| 每日摘要（vs 每条单推） | 避免推送轰炸 |
| 信源提名机制（用户推荐 → 编辑审核） | V1 先跑起来再开放策展 |
| 用户数据看板（自己看累计推了多少条） | 留存 hook |

### V3（3-6 个月）

**主题：商业化 + 规模化**

| 功能 | 为什么放 V3 |
|---|---|
| 付费（订阅制 20/月，免费版限每日 5 条） | 先要有需求验证（V2 后看用户量） |
| 邮件渠道 | 有用户明确要求后再做 |
| Web Console | 高级用户管理订阅 |
| 语义关键词匹配（embedding） | contains 够用 80%，embedding 收益边际 |
| 个性化推荐（基于点击反馈） | 需要积累行为数据 |

## §6 用户旅程

### 首次使用（Onboarding）

```
用户在朋友圈看到推荐 → 点开 bot 链接 → Telegram 打开
→ 发送 /start
→ Bot 回："欢迎使用 TrendRadar。发 /subscribe 关键词 开始订阅。推荐：AI、大模型、Agent、Arxiv"
→ 用户发 /subscribe agent 框架
→ Bot 确认订阅
→ 用户等待下次爬取（最多 1 小时）或发 /now 立即触发
```

### 日常（每天）

```
Bot 在用户订阅周期内推送匹配内容：

[07:30] [Arxiv] xxx 论文  🔗
[07:30] [GitHub] xxx 开源了 yyy  🔗
[12:00] [HN] xxx 讨论度高  🔗

用户点链接阅读，或划过。
```

### 管理订阅（TG 用户）

```
/list           → 列出我订阅了什么
/subscribe ai   → 新增
/unsubscribe 3  → 删除 id=3 的订阅
/pause          → 暂停推送（休假、周末可用）
/resume         → 恢复推送
```

（`/pause` / `/resume` 视工作量决定 V1 还是 V2。推荐 V1，成本极低，体验好。）

### §6.3 飞书群接入旅程（v0.3：TG 作为控制面）

**绑定（由群管理员在 TG 里操作）**
```
前置：该管理员在 TG 已经 /start 过 Bot（作为 TG 用户注册）
1) 群管理员在飞书里建群 + 加自定义机器人 + 拿到 webhook URL
2) 切到 TG，发给 Bot：
   /add_feishu_group <webhook_url> 大模型, agent
3) Bot 回：
   - 绑定成功，内部 group id = 42
   - 已发测试消息到群（群里同时收到欢迎卡片）
4) 群成员开始收到定时推送
```

**日常**
```
群里匹配到的新闻以富文本卡片形式出现：
  【分类】标题
  摘要第一段…
  [阅读原文]
```

**管理订阅（依然在 TG）**
```
/my_feishu_groups          → 列出 TG 用户名下的所有 Feishu 群（id / 群名 / 关键词 / 状态）
/edit_feishu_group 42 ai   → 替换 id=42 这条的关键词为 "ai"
/pause_feishu_group 42     → 暂停推送（保留记录）
/remove_feishu_group 42    → 彻底解绑，webhook 从 DB 移除
```

**v0.3 对比 v0.2 的变化**：
- 没有 Web form，没有 magic link，没有 Turnstile
- 绑定门槛：必须有 TG（V1 受众都在技术圈，可接受）
- 安全性提升：webhook URL 只经 HTTPS 传到 Bot 和 DB，不在 Feishu 群里明文传播

## §7 反向边界（明确不做）

| 不做 | 理由 |
|---|---|
| 通用新闻聚合 | 赛道太大，即刻 / Feedly 已占位；我们只做 AI 垂直 |
| 用户自添加任意 RSS URL | 降质风险大，V1 只有策展池 |
| 评论 / 转发 / 社交 | 不做内容社区，做信息管道 |
| 抓取付费墙内容（WSJ / FT / 信息） | 法律风险 + 无增值 |
| 原创内容生产（编辑点评、日报） | 不做媒体，做工具 |
| 一次性群发营销（Newsletter 式） | 每条按订阅过滤，不群发 |
| 本地 App（iOS / Android） | Telegram / 飞书本身就是 App，不重造轮子 |
| 任何 Web UI（v0.3 决策） | 控制面全放 TG，V2 若非技术用户接入再补 Web form |
| 飞书 Bot 应用（企业商店应用） | 需要企业认证流程长；V1 走自定义 webhook 够用 |

## §8 成功指标

### V1 成功标准（内测阶段）

| 指标 | 目标 |
|---|---|
| 邀请 5 名试用者跑 7 天（TG 3 + 飞书 2） | 至少 3 名每天有打开 |
| 推送点击率（点链接/收到） | ≥ 30% |
| 订阅关键词平均数/人 | ≥ 2 |
| 误推率（推了但用户划走） | ≤ 40%（contains 匹配的理论上限，用户侧打分） |
